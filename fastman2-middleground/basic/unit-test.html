单元测试

单元测试

为了让开发者更多的关注测试用例本身如何编写，而不是浪费时间在如何运行测试脚本等辅助工作上，框架对单元测试做了一些基本约定

基本约定

约定test目录为存放所有测试脚本的目录，测试脚本文件统一按${filename}.test.js来命名，必须以.test.js作为文件后缀。

一个应用的测试目录示例如下： 

test
  --controller
    --home.test.js
  --service
    --home.test.js

前后置任务

框架内部使用Mocha测试框架来构建单元测试，Mocha使用before/after/beforeEach/afterEach来处理前置后置任务。每个用例会按before -> beforeEach -> it -> afterEach -> after的顺序来执行，而且可以定义多个任务。

异步测试

支持多种不同的写法： 

// 使用返回 Promise 的方式
it('should redirect', () => {
  return app.httpRequest()
    .get('/')
    .expect(302);
});

// 使用 callback 的方式
it('should redirect', done => {
  app.httpRequest()
    .get('/')
    .expect(302, done);
});

// 使用 async（推荐）,因为可以在一个单元测试中做很多串行操作和业务编排
it('should redirect', async () => {
  await app.httpRequest()
    .get('/')
    .expect(302);
});

模拟请求

框架内部通过app.httpRequest()来模拟请求的发送，一个简单示例： 

it(`should a operation`, async () => {
  await app.httpRequest()
    .set('Accept', 'application/json')
    .post('/api/user')  // get、post、put都可以
    //.send({ name: 'john' })
    // .attach('avatar', 'test/fixtures/avatar.jpg')  // 也可以通过attach模拟上传图片
    .send('name=john')  // x-www-form-urlencoded upload（post请求可通过send发送请求参数）
    .expect('Content-Type', /json/)
    .expect('Content-Length', '15')
    // .expect(200)
    .expect((res) => {
      res.body.id = 'some fixed id';
      res.body.name = res.body.name.toLowerCase();
    })
    .expect(200, {
      id: '',
      name: 'join',
    })
    .end((err, res) => {
      if (err) throw err;
    })
});

expect()会通过输入参数的不同类型和个数来实现不同的构造函数，具体API如下：

.expect(status[, fn])

断言响应的状态码，即ctx.body.status（httpcode=200代表一次成功的响应）

.expect(status, body[, fn])

断言响应的状态码和内容，即ctx.body.status和ctx.body

.expect(body[, fn])

断言内容

.expect((res) => {})

通过自定义的断言函数来实现校验工作，如果校验失败可尝试抛出一个异常 

app.httpRequest()
  .get('/')
  .expect(hasPreviousAndNextKeys)

function hasPreviousAndNextKeys(res) {
  if (!('next' in res.body)) throw new Error("missing next key");
  if (!('prev' in res.body)) throw new Error("missing prev key");
}

Controller测试

Controller测试与Router有着密切的关系，因此开发者可以通过上面所说的模拟请求来发起一次真实的请求，将Router和Controller连接起来，而且还可以发送各种满足边界条件的请求数据，以测试Controller的参数校验完整性。

例如要给app/controller/home.js: 

// app/router.js
module.exports = app => {
  const { router, controller } = app;
  router.get('homepage', '/', controller.home.index);
};

// app/controller/home.js
class HomeController extends Controller {
  async index() {
    this.ctx.body = 'hello fastman2';
  }
}

写一个完整的单元测试，它的测试代码test/controller/home.test.js如下： 

const {
  app,
  assert,
  mock,
} = require('fastman2-middleground-framework').Test;

describe('test/controller/home.test.js', () => {
  describe('GET /', () => {
    beforeEach(() => {
      app.mockCsrf(); // 框架内部进行了CSRF防范，因此模拟时需要模拟该步骤，否则会抛出403
    });
    it('should status 200 and get the body', () => {
      // 对 app 发起 `GET /` 请求
      return app.httpRequest()
        .get('/')
        .expect(200) // 期望返回 status 200
        .expect('hello fastman2'); // 期望 body 是 hello fastman2
    });

    it('should send multi requests', async () => {
      // 使用 generator function 方式写测试用例，可以在一个用例中串行发起多次请求
      await app.httpRequest()
        .get('/')
        .expect(200) // 期望返回 status 200
        .expect('hello fastman2'); // 期望 body 是 hello fastman2

      // 再请求一次
      const result = await app.httpRequest()
        .get('/')
        .expect(200)
        .expect('hello fastman2');

      // 也可以使用内置函数assert来这样验证
      assert(result.status === 200);
    });
  });
});

Service测试

Service测试比较简单，只需要创建ctx即可拿到Service实例，然后调用Service方法即可。 

// app/service/user.js
class UserService extends Service {
  async get(name) {
    return await userDatabase.get(name);
  }
}

编写单元测试： 

describe('get()', () => {
  it('should get exists user', async () => {
    // 创建 ctx
    const ctx = app.mockContext();
    // 通过 ctx 访问到 service.user
    const user = await ctx.service.user.get('crixusshen');
    assert(user);
    assert(user.name === 'crixusshen');
  });

  it('should get null when user not exists', async () => {
    const ctx = app.mockContext();
    const user = await ctx.service.user.get('no man');
    assert(!user);
  });
});

Mock方法

除了上面介绍过的app.mockContext()和app.mockCsrf()方法外，还提供了其他便捷的Mock方法来方便编写单元测试。

如想摸你一次请求的Session数据，可通过app.mockSession(data)来模拟。 

describe('GET /session', () => {
  it('should mock session work', () => {
    app.mockSession({
      foo: 'bar',
      uid: 123,
    });
    return app.httpRequest()
      .get('/session')
      .expect(200)
      .expect({
        session: {
          foo: 'bar',
          uid: 123,
        },
      });
  });
});

由于mock之后会一直生效，为了避免污染后面的单元测试，因此我们需要通过钩子函数afterEach执行一些清理工作。 

describe('some test', () => {
  // before hook

  afterEach(mock.restore);

  // it tests
});

虽然你已经知道了上面的约束，但是框架已经很好的将mock的清理工作封装掉了，因此上面的代码你不需要自行实现。

Mock属性和方法

在某些场景下，例如我们想编写一个Controller的单元测试，但是它却依赖了Service的某个函数，由于我们只是测试Controller，因此Service我们可以通过Mock来模拟。

Mock Service
app.mockService('user', 'get', () => {
    return {
      name: 'fengmk1',
    };
  });
Mock某个对象的方法
mock(fs, 'readFileSync', filename => {
  return 'hello world';
});
assert(fs.readFileSync('foo.txt') === 'hello world');
Mock HttpClient
// app.mockHttpclient(url, method, data)

// app/controller/home.js:
class HomeController extends Controller {
  async httpclient () {
    const res = await this.ctx.curl('https://api/gateway');
    this.ctx.body = res.data.toString();
  }
}

describe('GET /httpclient', () => {
  it('should mock httpclient response', () => {
    app.mockHttpclient('https://api/gateway', {
      // 模拟的参数，可以是 buffer / string / json，
      // 都会转换成 buffer
      // 按照请求时的 options.dataType 来做对应的转换
      data: 'mock gateway response',
    });
    return app.httpRequest()
      .get('/httpclient')
      .expect('mock gateway response');
  });
});
