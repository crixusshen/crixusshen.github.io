业务异常处理

业务异常处理

对运行时异常和业务异常进行统一抛出处理，开发者并不需要在可能出现异常的地方间接包裹try...catch进行处理。

异常处理可以说是整个framework相对比较复杂的模块，如果在底层不进行统一的考虑和封装，对于开发者将会是非常痛苦的开发过程。开发者可能会在需要进行异常处理的地方进行额外的try...catch处理，长此以往想象下整个逻辑代码会出现多少的try...catch。为了避免大量的异常捕获，我重新实现了整个异常处理的机制，通过灵活的配置来实现整个异常捕获，也同时考虑了更多的定制化和灵活性。

为了保证异常可追踪，在架构设计的时候就必须保证所有抛出的异常都是Error类型，因为只有Error类型才会带上堆栈信息，定位到问题。

关于config/errorCode.js文件声明

下面将这个文件统一定义为错误码声明文件，强烈建议开发者在这里统一定义错误码声明，然后在抛出异常的地方指定该错误码，下面是文件内定义错误码对象的格式和各字段说明： 

module.exports = {
  SYSTEM_EXCEPTION: {
    status: 500,
    message: 'System Exception',
  },
  USER_NOT_EXIST: {
    status: 400, // 响应的http状态码，如果未指定则统一以500服务器异常来表示
    code: '400', // 自定义的业务错误码，如果未定义该字段，则会使用`USER_NOT_EXIST`来表示
    message: 'can`t find user info', // 异常描述，建议合理定义，语义清晰，该值可能会呈现给具体用户
    errorPageUrl: '', // 如果http请求头字段accepts=text/html，则一旦定义该字段将会重定向到该地址
    addtion1: 'a', // 额外参数，可以定义多个，这将在最终抛出异常时一起携带，某些情况我们可能希望这样做，方便进行问题排查
  },
};

关于status的正确定义，开发者可从List of HTTP status codes进行查询。

API

为了让开发者使用简单，框架内部提供了4种API方式来扩展异常处理的过程，接下去会展开进行详细说明：

1. 抛出业务异常

对应API：ctx.throwBizError(code, error, bizParams)

  • code：error.code，如果未定义默认输出SYSTEM_EXCEPTION
  • error：可定义错误描述或一个Error类型对象
  • bizParams：额外的业务参数，能帮助开发者解决和排查问题
  • bizParams.sendClient：对象类型，定义后将使用errors包裹为对象，发送到客户端，例如下面示例中的{ userId: 1 }
  • bizParams.code：定义后将会覆盖error.code
  • bizParams.log：一旦定义将会打印日志到日志文件内，如果设置为false则不会打印日志文件

根据不同的入参顺序，框架会使用不同的构造函数来构建不同的异常处理需求。但是最终会抛出一个Error类型的异常对象，这个对象将会有以下几部分来组成： 

// Error Type Object
{
  // 传入的code，如果config/errorCode.js中定义了code，则将会覆盖传入的code，如果没有传入code则默认显示SYSTEM_EXCEPTION
  code: '',
  // 异常描述，可在config/errorCode.js中进行定义
  message: '',
  // 是否是业务异常
  bizError: true,
  // 业务相关字段，可方便排查问题，该字段如希望在生产环境在消息体中暴露，需要将config配置文件中的sendClientAllParams设置为true，默认情况下sendClientAllParams将被设置为false，无论该值如何设置，在测试和本地环境下都将输出开发者所定义的bizParams参数（被errors所包裹）
  bizParams: { /*...*/ },
  // 堆栈，可跟踪到具体的报错位置
  stack: '',
}

下面展示了一些输入参数的顺序调用： 

// 纯字符串，对应错误码声明文件中定义的错误码对象
ctx.throwBizError('SYSTEM_EXCEPTION');

// 等同于ctx.throwBizError('SYSTEM_EXCEPTION', { log: false });
ctx.throwBizError({ code: 'SYSTEM_EXCEPTION', log: false });

// 其中userId:1将会被errors包裹作为Error类型对象输出
ctx.throwBizError('SYSTEM_EXCEPTION', { userId: 1, log: false });

// 如果code在错误码声明文件中未找到则使用默认，如果找到但是错误码声明文件中未定义message，则以error输入参数为准
ctx.throwBizError('SYSTEM_EXCEPTION', 'error message');

// 等价于ctx.throwBizError('SYSTEM_EXCEPTION', 'xxxxxx');
ctx.throwBizError('SYSTEM_EXCEPTION', new Error('xxxxxx'));
2. 处理业务异常

对应API：ctx.responseBizError(error, bizParams)

一般在try...catch的catch中捕获业务异常后调用该API，框架内部已经实现了这个函数的调用，因此这里不再解释。

3. 监听业务异常

对应API：app.on('responseBizError', (ctx, error) => {})

开发者可以在app.js文件中进行业务异常的监听，当config中的配置参数interceptAllError=true时该监听事件才会被触发，否则不会触发，系统默认情况下不会监听该事件，即interceptAllError=false。

4. 业务异常处理器

对应API：app.BizErrorHandler

开发者可在app.js文件中重写默认的业务异常处理器，一般可生成自定义的消息结构。 

 async didLoad() {
    // 重写默认的业务异常处理器，一般可生成自定义的消息结构
    this.app.BizErrorHandler = class extends this.app.BizErrorHandler {
      json(ctx, error, config) {
        ctx.body = {
          code: config.code,
          msg: config.message,
        };
      }
    };
  }
